## （名词归类 + 环节定位 + 平行技术 + 常用搭配 + 前端适配插件，开发者一站式速查）

## 整体链路总览

![](.\img\1.png)



```
①摄像机采集成像 → ②硬件编码压缩 → ③视频存储/录像 → ④流媒体接入转码/协议转换
```

# 一、环节 1：摄像机【采集成像 + 硬件编码】

## 核心作用

光学图像 → 原始 YUV 裸视频 → 压缩编码成可传输存储的音视频码流

## 专业名词归类

- 硬件：IPC 网络摄像机、图像 Sensor、镜头、IR 红外补光
- 原始数据：**YUV/RGB**（未压缩裸流，体积超大）
- 编码标准：**H.264(AVC)、H.265(HEVC)、H.266**
- 码流概念：**主码流、子码流、三码流**、分辨率 (720P/1080P/4K)、帧率 FPS、码率 Bitrate
- 原生输出协议：**RTSP**（监控 IPC 标准输出）

## 平行技术 / 竞品

- 编码平行：H.264（兼容最强）↔ H.265（省 50% 带宽）↔ H.266（新一代高压缩）
- 输出协议平行：RTSP ↔ RTP/RTCP

## 常用搭配

IPC + H.264/H.265 双码流 + RTSP 输出

## 关联后续环节

RTSP 裸流送入**流媒体服务**处理

------

# 二、环节 2：视频【存储 / 录像】

## 核心作用

实时码流持久化，支持回看、取证、备份

## 专业名词归类

- 存储设备：**NVR 网络硬盘录像机、SD 卡、磁盘阵列、对象存储**
- 存储封装格式：MP4、FLV、TS、DAV（监控私有格式）
- 录像模式：定时录像、移动侦测录像、告警录像
- 回放相关：时间轴回放、倍速回放、切片录像

## 平行技术

本地 NVR 存储 ↔ 云端对象存储 ↔ 摄像头本地 SD 卡存储

## 常用搭配

IPC RTSP → NVR 接入存储 → 预分配硬盘配额

## 说明

实时预览链路**不依赖存储**，存储是独立录像分支

------

# 三、环节 3：流媒体服务【核心中转：拉流 + 转协议 + 分发】

## 核心作用

拉取 IPC RTSP 流 → 解复用 / 转封装 / 转协议 → 输出网页可播放流，是前后端桥梁

## 专业名词归类

- 主流流媒体框架：**ZLMediaKit、SRS、EasyDarwin、LiveNVR、Janus**
- 协议转换：RTSP → **HTTP-FLV / HLS(m3u8)**
- 能力：拉流、推流、转封装、时间切片、防盗链、并发分发
- 鉴权联动：对接 VMS 业务平台，生成带 Token 私有播放 URL

## 平行输出协议（关键）

| 输出流类型 |       特点        |       适用端        |
| :--------: | :---------------: | :-----------------: |
|  HTTP-FLV  |    低延迟 1~3s    |       PC 网页       |
| HLS(m3u8)  | 高兼容 3~15s 延迟 | iOS / 安卓 / 全终端 |

## 常用搭配

流媒体服务 + RTSP 入流 + 输出 HTTP-FLV/HLS + CDN 分发

## 关联前端

输出`.flv`/`.m3u8`地址，供前端拉播

------

# 四、环节 4：业务 VMS 平台【鉴权 + 管理 + 接口调度】

## 核心作用

用户登录、Token 鉴权、保活、获取播放地址、PTZ 云台控制、设备管理

## 专业名词归类

- 接口能力：登录接口、Token 令牌、心跳保活、播放地址下发、PTZ 云台控制
- 安全：防盗链、时效 URL、权限绑定 VID 摄像头 ID

## 常用搭配

VMS 后端 + axios 前端请求 + 长效 Token 保活

## 对应你代码

`btnLogin()`/`btnGetURL()`/`ptzCtrl()` 全部属于该环节

------

# 五、环节 5：网络传输层

## 专业名词归类

- 传输基础：HTTP/TCP、CDN、内网 / 公网、带宽抖动、丢包
- 流媒体传输：HTTP 长连接 (FLV)、HLS 分片 HTTP 请求

## 平行传输对比

- HTTP-FLV：流式长连接，低延迟
- HLS：分片短轮询，高兼容

## 影响体验

带宽不足 → 卡顿 → 前端需追帧 / 切换子码流

------

# 六、环节 6：前端【解封装 + 播放控制】

## 核心拆分：解封装 ≠ 解码

### 6.1 解封装层（JS 库处理）

#### 专业名词 & 插件匹配

- 播放 FLV 流必备：**flv.js**
- 播放 HLS 流必备：**hls.js / video.js**
- 关键配置：`enableWorker:true`（WebWorker 后台解封装，不阻塞主线程）
- 直播优化：缓冲控制、**追帧代码**（倍速 / 跳转延时校准）

#### 平行前端插件

flv.js（HTTP-FLV 专用）↔ hls.js（HLS 专用）

### 6.2 浏览器解码 & 渲染层

#### 专业名词

**浏览器硬解码 (GPU 解码)**、HTML5 `<video>`标签、H.264/H.265 解码兼容性

- flv.js/hls.js 只做**解封装剥壳**，不解码
- 真正画面解码：浏览器调用 GPU 硬解

## 该环节常用搭配

HTTP-FLV 流 + flv.js + enableWorker:true + 自定义定时追帧 + 浏览器硬解

## 对应你代码全播放逻辑

createFlvPlayer → load () → play () → 2s 间隔追帧控延迟

1. **编码看摄像头**：H.264 兼容、H.265 省带宽；主码流高清录像，子码流网传预览
2. **中转看流媒体**：RTSP 转 HTTP-FLV 低延迟、转 HLS 全兼容

# 其他总结



## 一、264 265 和RTSP关系

三者最标准的关系：

```
IPC摄像头采集画面
↓
H.264/H.265 硬件编码（把画面压缩）
↓
RTSP 协议传输（把压缩好的码流送出去）
```

### 1. H.264 / H.265（编码格式）

- **本质**：视频的**压缩算法**，负责把摄像头采集的超大体积原始 YUV 画面，压缩成可传输、可存储的小体积码流。
- **作用**：决定视频的**画质、体积、带宽占用、兼容性**。
- **类比**：就像把一个大文件**压缩成 ZIP 包**，H.264 是老压缩算法，H.265 是新压缩算法（同画质体积小一半）。

### 2. RTSP（实时流协议，Real Time Streaming Protocol）

- **本质**：视频的**传输控制协议**，负责在网络上「建立连接、控制播放、传输视频流」。
- **作用**：是摄像头和流媒体服务之间的**标准通信协议**，用来拉取摄像头的实时画面。
- **类比**：就像一条**运输管道**，负责把压缩好的视频包，从摄像头送到流媒体服务器。

  

   摄像机拍照到视频，视频是变化的图片，图片按照何种的裁切手段（H.264/H.265），以何种手段（RTSP）发送到流媒体服务器。

H.264/H.265，发生在摄像头的内部，RTSP发生在摄像头输出阶段。

H.264/H.265是内容，小船；RTSP：是河流，载体。

总结：

1.RTSP 是传输协议，不是编码，和 H.264/H.265 完全不是一回事。

2.H.265 可以用 RTSP、HTTP-FLV、HLS 等任何协议传输，RTSP 只是摄像头的原生输出协议。

3.RTSP 是「管道」，H.264 是「管道里的货物」，管道可以装不同的货物。

### 一句话终极总结

- **H.264/H.265 = 视频的「内容 / 格式」，负责把画面变小**

- **RTSP = 视频的「运输协议」，负责把压缩好的画面从摄像头送出去**

- 二者是**内容与载体**的关系，共同构成了摄像头的标准输出链路，是安防监控的基础技术组合。

  

二者是「内容」和「载体」的关系

| 维度               | H.264 / H.265                   | RTSP                           |
| ------------------ | ------------------------------- | ------------------------------ |
| **层级**           | 应用层的「编码层」              | 应用层的「传输控制层」         |
| **核心作用**       | 压缩视频，决定画质 / 体积       | 建立连接，传输视频流           |
| **是否独立存在**   | 可以独立存在（如本地 MP4 文件） | 必须承载编码后的视频流才能工作 |
| **在链路中的位置** | 摄像头内部，编码环节            | 摄像头输出，网络传输环节       |



## 二、什么是浏览器硬解码？

### 定义

浏览器调用**电脑 / 手机 GPU 专用视频解码硬件模块**，并行解压 H.264/H.265 码流，输出画面渲染；

不靠 CPU 软计算解码，叫**硬解码**。

### 对比：硬解码 vs 软解码

| 维度     | 浏览器硬解码 (GPU)    | 软解码 (CPU JS 解码)   |
| -------- | --------------------- | ---------------------- |
| 占用资源 | CPU 占用极低、流畅    | CPU 飙升、卡顿、发热   |
| 性能     | 4K / 多路高清轻松跑   | 只适合低分辨率，多路崩 |
| 功耗     | 省电、设备不发烫      | 耗电高、笔记本降频     |
| 你的业务 | flv.js + 浏览器硬解 ✅ | 不使用                 |

## 结合你项目的完整链路（硬解码位置）

```
IPC → H.264/H.265编码
→ 流媒体转 HTTP-FLV
→ 网络传输FLV流
→ flv.js【解封装剥壳】
→ 输出原生H.264/H.265 ES码流
→ 浏览器递交【GPU硬件解码器】← 这里就是硬解码
→ 解码成画面帧 → <video>渲染播放
```

## H.264 / H.265 浏览器硬解码兼容性关键点

### 1. H.264

✅ **全浏览器、全显卡、全系统原生硬解码支持**

- 兼容性天花板，你的项目兜底首选

### 2. H.265(HEVC)

⚠️ 授权 + 硬件双重限制：

- Chrome / Edge 新版部分支持；
- Safari 支持好；
- 老旧显卡、低端工控机、部分浏览器**无 H.265 硬解码**；
- 无硬解码时：要么黑屏、要么浏览器自动降级软解→CPU 爆满卡顿

> 你代码里 `hasAudio:false` 监控无音频，只靠视频硬解码输出画面

## 硬解码对安防监控的核心价值

1. **多路同时预览不卡顿**（厂区 / 大屏轮训多路摄像头）
2. **低配电脑 / 工控机也能流畅播放**
3. 配合你代码里的「**追帧逻辑**」：硬解低延迟 + 前端追帧，整体延迟 1~3s
4. 降低设备发热、浏览器不掉帧、不崩溃

## 常见问题 & 排查点

1. 播放黑屏、有声音无画面

   

   → 大概率：H.265 当前浏览器 / 显卡

   不支持硬解码

2. CPU 占用很高、画面卡顿

   

   → 硬解码失效，浏览器被迫 CPU 软解

3. 切换子码流流畅，主码流 4K 卡顿

   

   → 显卡 H.265 硬解能力不足

4. flv.js 配置 

   ```
   enableWorker:true
   ```

   

   → 解封装放 WebWorker，不阻塞主线程，

   配合硬解更流畅

## 一句话总结

> 浏览器硬解码 = **GPU 专用硬件解压视频码流**
>
> flv.js 只负责拆开 FLV 外壳，真正把码流变成看得见画面的，是**浏览器 GPU 硬解码**；
>
> H.264 通吃所有设备硬解，H.265 画质好省带宽，但受硬件 + 浏览器授权限制。

## 三、子码流，主码流 解读

## 核心定义（IPC 摄像头双码流技术）

安防摄像头普遍支持**双码流编码输出**：

### 1. 主码流（Main Stream）

- 高分辨率、高码率、高画质
- 常用：4MP/8MP、1080P/4K、H.264/H.265
- 用途：**本地录像存储、高清回放、大屏上墙预览**
- 带宽消耗大，网络差容易卡顿

### 2. 子码流（Sub Stream）

- 低分辨率、低码率、压缩高
- 常用：360P/480P、小画幅
- 用途：**手机预览、网页多路轮播、窄带宽远程查看**
- 带宽小、流畅、省流量

## 、关键参数对比

| 维度       | 主码流             | 子码流                        |
| ---------- | ------------------ | ----------------------------- |
| 分辨率     | 高 (1080P/4K)      | 低 (360P~720P)                |
| 码率       | 大，占用带宽高     | 小，占用带宽低                |
| 画质       | 清晰、细节多       | 一般、够用                    |
| 网络要求   | 专线 / 内网高带宽  | 外网 / 4G / 弱网都能用        |
| 典型场景   | 录像取证、高清查看 | 网页多路预览、手机 APP 实时看 |
| 你页面参数 | `idx=1`            | `idx=2`                       |

## 和你代码精准对应

```
// 主码流：idx=1
<input id="idx" value="1" placeholder="1主码流，2子码流">
// 子码流：idx=2

// 获取播放URL核心拼接
"/vmslive?vid=xxx&idx=1&cmd=play&type=flv&token=xxx"
```

`idx=1` → 请求摄像头**主码流 FLV 流**，高清、吃带宽

`idx=2` → 请求摄像头**子码流 FLV 流**，流畅、省带宽

## 业务场景选型建议

1. **内网高清单路查看、录像** → 选 `主码流(idx=1)`
2. **外网远程访问、网页多窗口同时播放、弱网络** → 选 `子码流(idx=2)`
3. 主码流卡顿 / 黑屏 → 切换子码流立刻流畅

## 结合全播放链路

```
IPC 同时编码输出：主码流+H.265 / 子码流+H.264
↓
流媒体服务分别转发两路 HTTP-FLV
↓
前端传 idx=1/idx=2 拿到对应播放地址
↓
flv.js解封装 → 浏览器硬解码播放
```

## 常见问题

- 外网主码流播放卡顿、延迟高：**切子码流**解决
- 子码流看不清细节：内网环境切回主码流
- H.265 主码流部分浏览器不硬解黑屏：换子码流 H.264 兼容更好

## 一句话总结

- **主码流 idx=1**：高清大带宽，存录像、精细查看

- 子码流 idx=2

  ：流畅小带宽，网页 / 手机弱网预览

  是安防监控为「不同网络、不同场景」设计的双路独立视频码流。

## 四、H.264/H.265 对比分析

核心定义一句话总结

- **H.264（AVC）**：2003 年发布的**经典视频编码标准**，全平台兼容，安防 / 直播 / 视频行业的「通用兜底方案」。
- **H.265（HEVC）**：2013 年发布的**新一代高效编码标准**，在 H.264 基础上压缩率提升 50%，同画质体积减半，4K / 超高清场景首选。

核心参数对比表（开发选型速查）

| 对比维度            | H.264（AVC）                                                 | H.265（HEVC）                                                |
| ------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **压缩效率**        | 基准水平，同画质体积大                                       | 压缩率提升**50%**，同画质体积仅为 H.264 的 50%               |
| **带宽占用**        | 高，1080P 直播通常需 4-8Mbps                                 | 低，1080P 仅需 2-4Mbps，4K/8K 超高清优势显著                 |
| **编码 / 解码算力** | 低，软硬解码广泛支持，老旧设备无压力                         | 高，对 CPU/GPU 硬件要求高，老旧设备不兼容                    |
| **兼容性**          | **全平台天花板**：所有浏览器、手机、摄像头、播放器 100% 支持 | 部分受限：老旧安卓、低端工控机、部分浏览器 / 显卡不支持硬解码 |
| **分辨率支持**      | 最高 4K，主流 1080P                                          | 最高 8K，完美支持 4K/8K 超高清                               |
| **专利授权**        | 专利池费用低，商用成本低                                     | 专利授权复杂，部分厂商需缴纳高额专利费                       |
| **适用场景**        | 标清 / 高清直播、全平台兼容、老旧设备、弱网兜底              | 4K / 超高清监控、省带宽场景、高端设备、新系统部署            |
| **安防监控适配**    | 子码流 / 外网预览首选，全终端兼容                            | 主码流 / 内网高清存储首选，省存储空间                        |
| **浏览器硬解码**    | 全浏览器原生支持，无黑屏风险                                 | Chrome/Edge 新版支持，部分浏览器 / 显卡无硬解会黑屏 / 卡顿   |

## 核心技术差异深度解析

### 1. 压缩效率（核心优势）

H.265 通过**更大的编码块（CTU）、更灵活的帧间预测、改进的环路滤波**等技术，实现了**同画质下体积减半**：

- 1080P 视频：H.264 需 4Mbps，H.265 仅需 2Mbps，带宽节省 50%
- 4K 视频：H.264 需 16Mbps，H.265 仅需 8Mbps，存储 / 带宽成本直接减半
- 结论：H.265 是**带宽 / 存储紧张场景的最优解**

### 2. 算力与性能要求

- **H.264**：编码 / 解码逻辑简单，摄像头、手机、电脑的 CPU/GPU 都能轻松硬解，多路同时播放无压力
- **H.265**：编码复杂度提升约 2-3 倍，解码复杂度提升约 1.5-2 倍，老旧设备（如 5 年前的安卓手机、低端工控机）会出现**软解 CPU 爆满、卡顿、发热**，甚至无法硬解黑屏

### 3. 兼容性（选型关键）

- **H.264**：从 2003 年沿用至今，是所有视频设备的「基础标配」，无任何兼容性问题，是项目兜底的唯一选择
- **H.265**：受专利和硬件限制，部分老旧浏览器、低端摄像头、工控机不支持，iOS/Safari 支持较好，安卓 / Chrome 需新版系统 / 浏览器

## 结合你项目（安防监控 + 网页播放）的选型建议

### 1. 主码流 / 内网场景

✅ **优先选 H.265**

- 优势：4K/1080P 高清存储，省 50% 硬盘空间；内网带宽充足，硬解码无压力
- 适用：本地录像、大屏上墙、内网单路高清预览

### 2. 子码流 / 外网 / 网页全平台场景

✅ **优先选 H.264**

- 优势：全浏览器 / 手机兼容，无黑屏风险；外网弱网下流畅不卡顿
- 适用：网页多路预览、手机 APP 远程查看、外网访问

### 3. 双码流黄金搭配（安防行业标准方案）

摄像头同时输出两路流：

- **主码流：H.265** → 用于本地高清存储、内网大屏
- **子码流：H.264** → 用于网页 / 手机远程预览、外网访问
- 完美平衡「画质 / 存储 / 带宽 / 兼容性」，是你项目的最优架构

## 常见问题与避坑指南

### 1. 为什么 H.265 播放黑屏 / 卡顿？

- 核心原因：**浏览器 / 显卡不支持 H.265 硬解码**，被迫 CPU 软解，导致卡顿 / 黑屏
- 解决方案：切换 H.264 子码流，或升级硬件 / 浏览器

### 2. H.265 能完全替代 H.264 吗？

- 不能！H.264 的**全平台兼容性**是不可替代的，H.265 仅适合新设备 / 新系统，老旧项目必须用 H.264 兜底

### 3. 网页播放 H.265 需要注意什么？

- 必须依赖**浏览器硬解码**，flv.js 仅做解封装，不负责解码
- 开启`enableWorker:true`，避免解封装阻塞主线程，配合硬解码更流畅
- 需做兼容性降级：检测到不支持 H.265 时，自动切换 H.264 流

## 一句话终极总结

- **H.264 = 全平台兼容、低算力、通用兜底，所有场景都能用**
- **H.265 = 高压缩、省带宽、超高清，新设备 / 内网 / 存储场景首选**
- 安防监控最佳实践：**主码流 H.265 存高清，子码流 H.264 做预览**，兼顾画质、成本与兼容性

### flv hls  和m3u8 ts区别

一句话终极区分

**HLS = 协议（播放方案）**

**m3u8 = 索引文件（清单）**

**ts = 视频切片文件（货物）**

**FLV = 另一种独立的视频封装格式（流式）**

```
【HLS 协议】
       ↓
由两个东西组成：
   1. m3u8（清单文件，告诉播放器播哪段）
   2. ts（视频切片，真正的画面内容）
```

【FLV】   独立格式，单个长流，不分片

## 1. **HLS**

- 苹果推出的**直播 / 点播协议**
- 原理：把视频切成一小段一小段
- **延迟高（3~15 秒）**
- **全平台兼容（iOS / 安卓 / 浏览器全支持）**

## 2. **m3u8**

- **HLS 的清单文件**
- 文本文件，里面列了一堆 `.ts` 文件名
- 播放器先下载 m3u8 → 知道要下载哪些 ts
- 你看到的播放地址：`xxx.m3u8`

## 3. **ts**

- **HLS 里的视频切片文件**
- 每片 2~5 秒
- 里面是真正的 H.264/H.265 画面
- 一堆 ts + 一个 m3u8 = 一套 HLS

## 4. **FLV**

- **流式长视频格式**
- 不分片，一整流一直传
- **延迟极低（1~3 秒）**
- 必须用 flv.js 播放
- **iOS 浏览器不支持**
